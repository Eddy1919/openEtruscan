# Plan — CV detector properly trained on Etruscan glyphs

**Status:** plan only. The existing [`src/cv_pipeline/train_yolo.py`](../../src/cv_pipeline/train_yolo.py) is a thin wrapper over `ultralytics` that points at synthetic data; this document specifies the corpus, label policy, model, training schedule, eval metric, and deployment surface for a *real* trained detector that's worth shipping.

## What "properly trained" means here

The existing pipeline at [`src/cv_pipeline/`](../../src/cv_pipeline/) trains YOLOv11-nano on **synthetic** Etruscan-glyph images generated by [`generate_synthetic_data.py`](../../src/cv_pipeline/generate_synthetic_data.py) — a font-rasterisation routine that draws Old-Italic Unicode glyphs (`U+10300..U+1032F`) onto plain backgrounds with random crops, rotations, and lighting. This gets a model that detects clean Old-Italic Unicode shapes; it does **not** generalise to actual Etruscan epigraphy because real inscriptions are:

- **Stone, bronze, ceramic** — not vector font;
- **Worn, broken, retouched** by 2,000 years of weathering;
- **Stylistically variant** — Cerveteri vs Tarquinii hands differ as much as ninth-century Caroline minuscule differs from twelfth-century Gothic;
- **Mixed-direction** — boustrophedon, right-to-left, left-to-right within the same corpus;
- **Sometimes carved over each other** (palimpsests on the same support).

A "properly trained" detector means one whose **training distribution actually contains real Etruscan epigraphic surfaces**, with labels that survive expert philological review, evaluated on a **held-out site or repository** so we know it generalises beyond the seen Etrurian centres.

## Dataset — what to collect

### Primary source: the 6,633 unified inscriptions we already have

Every row in the `inscriptions` table has at least one of: `image_url`, `findspot`, `medium`, `object_type`. About 2,317 (the `acquired_documented` provenance tier) have a named findspot suitable for spatial citation. Of those, our best guess is **~600–800 inscriptions** that have *published images* in the source bibliographies or in the Beazley Archive / Trismegistos image set we can resolve via the Pelagios graph integrations already shipped. Concretely:

1. **CIE Vol. I extractions** — many inscriptions ship with a line-drawing rendering in the published volume. The line drawings are NOT what we want (they're scholar-interpretive, not direct photography), but they DO give us bounding boxes for free.

2. **Trismegistos cross-links** — every `trismegistos_id` row can be resolved to a TM-Texts record that may have an attached photograph. We already have 1,847 such resolved IDs.

3. **EAGLE federated photographs** — EAGLE aggregates ~25k epigraphic photographs across European archives, of which ~3,200 are tagged "Etruscan". Available under the EAGLE LOD endpoint that the Pelagios integration can reach.

4. **Pleiades-aligned site photos** — 184 findspots resolve to Pleiades places; about 90 of those have associated photographs in the Pleiades partner archives (DAI Rome, BSR, ICCD).

5. **User-contributed** (longer-tail) — frontend `/contribute` page (separate work) where philologists can upload high-resolution photographs of inscriptions with already-public bibliography. CC-BY-licensed.

Conservative target for the v1 detector training set: **800 inscription photographs** with **~25,000 glyph-level bounding boxes**. This is small by COCO standards but large by epigraphic-CV standards (current published Etruscan-glyph datasets are < 100 inscriptions).

### Label policy — what's a class?

The single hardest choice. Three plausible label schemes:

**(scheme A) — 26 Old-Italic letter classes.**
One class per Unicode Old-Italic code point `U+10300..U+1031F`. Conceptually clean; matches the existing `normalizer` system's output. *Drawback:* doesn't distinguish stylistic variants that matter philologically (e.g. the Vetulonian sigma vs the Volsinian sigma).

**(scheme B) — 26 letters × per-script-region variants ≈ 70–90 classes.**
Adds variants for each of the major Etruscan epigraphic centres (Cerveteri, Tarquinii, Vulci, Volsinii, etc.). *Drawback:* extreme class imbalance — some variant cells will have < 5 examples in the dataset.

**(scheme C) — Letter detection only (no class), with classification deferred to a second pass.**
Detector emits "this region is a glyph"; a separate CharCNN classifier (already shipped in [`classifier.py`](../../src/openetruscan/ml/classifier.py)) reads the cropped patch and emits letter + variant. *Drawback:* loses joint information between location and identity; the detector can't use letter shape to refine its bounding box.

**Recommendation: start with scheme A (26 classes), revisit if precision plateaus.** Scheme C is also defensible and is what most production OCR systems do. Avoid scheme B until we have ≥ 50 examples per variant cell.

### Label provenance

Bounding boxes come from three sources, in descending trust:

1. **Hand-labelled** by a philologist via [Label Studio](https://labelstud.io/) — the [`convert_to_ls_json.py`](../../src/cv_pipeline/convert_to_ls_json.py) export already exists, the import path needs the photograph URL list. ~500 inscriptions × ~30 glyphs = ~15,000 boxes. Time: ~80 hours at 10 sec/box.

2. **Pseudo-labelled** by an iterated v0 detector — train v0 on the hand-labelled 500, run on the remaining ~300 inscriptions, take top-confidence predictions as new training data, retrain. The existing [`yolo_pseudo_labeler.py`](../../src/cv_pipeline/yolo_pseudo_labeler.py) is set up for this loop. Quality control: hand-review 5% of the pseudo-labels per round.

3. **Font-similarity-bootstrapped** — the existing [`font_similarity_labeler.py`](../../src/cv_pipeline/font_similarity_labeler.py) compares image patches to Old-Italic Unicode font renderings via deep-feature cosine; bounding boxes are derived from edge-detection + grid alignment. Lowest trust; useful only for the very first v0 bootstrap before any hand-labels exist.

## Model — what to actually train

### v0 (existing): YOLOv11-nano on synthetic + font-similarity-labelled

Current state. ~5.4 MB weights. Trains in ~2 hours on a T4. Doesn't generalise to real photographs.

**Disposition:** keep as the bootstrap labeller for v1 (drives `yolo_pseudo_labeler.py`), don't ship to users.

### v1 (target): YOLOv11-small on hand-labelled + pseudo-labelled real photographs

**Why YOLOv11-small (not nano):** real-photograph Etruscan glyphs are *much harder* than synthetic font ones. Weathering, lighting, scratches all reduce signal-to-noise. The ~5 M extra parameters in `-s` over `-n` are worth it on the eval distribution; we lose some FPS on edge inference but gain meaningful mAP. (If edge-FPS becomes critical later, distill `-s` → `-n`.)

**Training schedule:**

| Phase | Data | Epochs | LR | Augmentation | Output |
|---|---|---:|---|---|---|
| v0 bootstrap | synthetic font + font-sim labels | 100 | 1e-3 | mixup, mosaic, hsv | weights for v0 only |
| v1 base | 500 hand-labelled photos | 200 | 5e-4 → 1e-5 cosine | mixup, mosaic, hsv, rotate ±10°, perspective | v1-base |
| v1 pseudo | + ~300 pseudo-labelled (v1-base confidence ≥ 0.7) | 100 | 1e-4 | same | v1-final |

**Training compute:** ~6 hours total on a single T4 (n1-standard-8 Vertex job). **~$2 spend, billed to `double-runway-465420-h9`.**

**Augmentation note:** Etruscan epigraphy includes boustrophedon writing — text alternating direction line-by-line. We **must NOT** include horizontal flip in augmentation (would teach the model that `A` and reversed-`A` are the same, which they're not in our 26-class scheme — Old Italic mirror-variants are separate code points). Rotation ±10° only.

### v2 (stretch goal): YOLOv11-small + script-direction head

Auxiliary task: predict the line's writing direction (LTR / RTL / boustrophedon-LTR / boustrophedon-RTL). Trained jointly with detection via a small MLP head on the detector backbone. Helps downstream OCR sequencing.

Cost: minimal — same training run, +20% compute for the extra head. Worth doing iff the line-direction labels come for free from the bibliography (most published inscriptions are tagged with direction).

## Evaluation — what counts as ≥ ship-ready

**Eval set:** 100 inscriptions from a **held-out site** — Cerveteri inscriptions if the training set is dominated by Tarquinii / Vulci / Volsinii, or vice versa. This is the only way to know we're not just memorising one school's stylistic conventions.

| Metric | v1 ship gate |
|---|---:|
| Glyph mAP@0.5 IoU | ≥ 0.65 |
| Per-class precision@0.7 confidence, averaged | ≥ 0.75 |
| Recall on **complete** inscriptions (every visible glyph detected) | ≥ 0.80 |
| Boustrophedon handling (no L/R flip errors) | 0 false-mirror detections in eval |
| Edge inference latency, 512×512, ONNX, browser-side | < 200 ms p50 |
| Onnx model size | < 30 MB |

**The 0.65 mAP gate is intentionally modest.** State-of-the-art YOLO models on COCO sit at 0.50-0.60 mAP@0.5 with much larger training sets and more diverse classes. Epigraphic CV in 2025 publishes papers at mAP@0.5 = 0.50-0.70 on similarly-sized datasets; 0.65 is realistic.

## Deployment

### Browser-side (primary surface)

The existing [`/classifier`](../../../openetruscan-frontend/app/classifier/) page already runs ONNX models client-side via [ONNX Runtime Web](https://onnxruntime.ai/docs/tutorials/web/). Same path:

1. Export the trained YOLO weights to ONNX (`yolo export format=onnx imgsz=512`).
2. Quantise to int8 if size > 30 MB.
3. Add to `public/models/glyph-detector-yolo.onnx`.
4. Add a new `/explorer/cv` page (or hook into the existing `/inscription/[id]/cv` viewer) that runs detection on any photograph the user uploads.

### Server-side (batch)

For the ingestion pipeline: a Cloud Run Job in `double-runway-465420-h9` that processes the EAGLE / Trismegistos federated photographs in bulk, writes per-image bounding-box JSON to GCS, joins back to `inscriptions.id` via the existing `trismegistos_id` field.

### HF Hub artefact

`openEtruscan/glyph-detector-yolo` — repo-id already configured in `train_yolo.py:25` (`HF_REPO_ID`). Model card lives in `models/glyph-detector-yolo/README.md` (will need to be created when v1 is ready to ship). Push helper analogous to [`scripts/hub/push_etr_lora_v4.py`](../../scripts/hub/push_etr_lora_v4.py).
